## How Can I Contribute?

Contributions and bug reports are tracked through **GitHub Issues**.

Before opening a new issue, please check the existing issues to avoid duplicates.

When contributing, follow these guidelines to help maintainers understand your report, reproduce the issue, and evaluate proposed changes:

- Contributions include **code** (scripts, Makefiles, Dockerfiles), **Quarto reports and outputs**, and **documentation**.  
- Make sure your local repository is **up-to-date with `main`** before beginning work.  
- Commit frequently with **clear and descriptive messages**.  

---

## Reporting Issues

If you are reporting a bug or problem, please include as many details as possible:

- Use a **clear and descriptive title**.  
- Describe the **steps required to reproduce the problem**.  
- Provide **examples** such as code snippets, screenshots, or links to files if relevant.  
- Explain the **expected behavior** and what actually occurred.  
- Include details about your **system or environment** if relevant.  

Clear issue descriptions help maintainers diagnose and resolve problems efficiently.

## Direct/Core Contributors

Direct contributors (project collaborators) should follow this workflow:

- Use **GitHub Issues** to track tasks and bugs.  
- Assign issues when work begins.  
- Link issues to project milestones where appropriate.  
- Pull the latest version of the repository before starting work.  
- Push updates regularly.  

All changes should follow a **GitHub Flow workflow**:

1. Create a new branch for each feature, fix, or update.  
2. Implement and test the changes locally.  
3. Commit frequently with **descriptive messages**.  
4. Push the branch to the repository.  
5. Open a pull request describing the update and link relevant issues.  
6. At least one team member reviews the changes.  
7. After approval, merge the pull request into `main`.  

> ✅ Tip: Always pull the latest `main` and resolve any conflicts before opening a PR.

## External Contributors

External contributors are welcome.

If you would like to contribute:

1. Open an issue describing the change or improvement you would like to make.  
2. Fork the repository.  
3. Create a branch for your modification.  
4. Implement your changes and commit with clear messages.  
5. Push the branch to your fork.  
6. Submit a pull request from your fork to the main repository.  

Project maintainers will review the pull request and provide feedback if needed.

> ⚠️ Note: External contributors should **never push directly** to the `main` branch in the original repository.

---

## Branch Naming Guidelines

Use **descriptive and consistent branch names**:

- Feature branches: `feature/<short-description>` or `<feature-name>`  
- Bugfix branches: `bugfix/<short-description>`  
- Documentation updates: `docs/<short-description>`  
- Hotfixes: `hotfix/<short-description>`  

Example: `feature/makefile-update` clearly indicates the purpose of the branch.

---

## Pull Requests and Reviews

- Provide a **clear title and description** of the changes.  
- Link the PR to **related issues or milestones**.  
- Include **screenshots or examples** if the change affects outputs.  
- At least one **peer review** is required before merging.  
- Resolve any **merge conflicts** promptly.  

> ✅ Tip: Use draft PRs if your work is still in progress but you want feedback early.
